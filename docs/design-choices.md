# Design Choices

This document explains what the generator intentionally does, what it does not do, and what that means when you are starting with this project.

## What This Generator Produces

The generator produces a small, Spring-focused client surface:

- `<basePackage>.client`: Java interfaces using Spring HTTP Interface annotations (`@GetExchange`, `@PostExchange`, etc.).
- `<basePackage>.models`: Java records, enums, and union interfaces for schema types.

Design goal: generate minimal code that is easy to read, replace, and integrate into an existing Spring application.

## Client Generation Choices

### Spring HTTP Interface style only

Generated clients are plain Spring HTTP interfaces. The generator does **not** create custom runtime wrappers around `RestClient` or `WebClient`.

### Supported HTTP methods

Only these methods are generated:

- `GET`
- `POST`
- `PUT`
- `DELETE`
- `PATCH`

### Operation grouping and naming

- Operations are grouped by OpenAPI tag into `*Api` interfaces.
- If an operation has no tags, it goes into `DefaultApi`.
- If `operationId` is present, it is used for method naming.
- If `operationId` is missing, a fallback name based on `method + path` is generated.
- Names are sanitized to valid Java identifiers.

### Parameters and request/response handling

- Supported parameter locations: `path`, `query`, `header`.
- Path-level and operation-level parameters are merged; operation-level definitions override path-level ones with the same `in + name` key.
- For request/response content, `application/json` is preferred; otherwise the first available media type is used.
- `application/octet-stream` with `type: string, format: binary` maps to Spring `Resource` for request bodies.

### 2xx response rule

Client return types are determined from 2xx responses:

- If exactly one 2xx response has a body, that body type is used.
- If multiple 2xx responses have bodies with the **same** schema, the first is used.
- If multiple 2xx responses have **different** body schemas, generation fails.
- If no 2xx response has a body, return type is `void`.

### Blocking and reactive variants

The serializer supports:

- `BLOCKING` mode (default)
- `REACTIVE` mode (`Mono<T>` / `Mono<ResponseEntity<T>>`)

For each operation, two methods are generated:

- regular return type (`T`/`void` or `Mono<T>`)
- `ResponseEntity` variant (`...ResponseEntity`)

## Model Generation Choices

### Records by default

Object schemas become Java records.

- Required fields get `@JsonProperty(required = true)` unless disabled by config.
- Optional fields are serialized with `JsonInclude.NON_NULL`.
- Required fields are serialized with `JsonInclude.ALWAYS`.

### Builders

Builders are generated by default (`STRICT` mode):

- `STRICT`: `build()` enforces non-null for fields that are both required and non-nullable.
- `RELAXED`: builder exists but does not enforce that non-null check.
- `DISABLED`: no builder is generated.

### Nullability and `NullWrapper<T>`

For fields that are both **optional** and **nullable**, generated models use `NullWrapper<T>`.

Reason: this preserves a three-state distinction that plain `@Nullable` cannot represent cleanly:

- field absent
- field present with `null`
- field present with value

`NullWrapper` serializer/deserializer code is generated only when needed.

### `additionalProperties`

Object schemas with `additionalProperties` generate an `additionalProperties` map (`Map<String, Object>`) with `@JsonAnyGetter/@JsonAnySetter`.

## Combined Schema Choices (`oneOf`, `anyOf`, `allOf`)

### `oneOf`

- `oneOf` maps to a generated union interface when there are multiple variants.
- Variant models implement that union interface.
- If a discriminator property is available and discriminator values can be inferred for all variants, Jackson polymorphism annotations are added.
- If discriminator values cannot be inferred consistently, discriminator annotations are omitted.

### `anyOf` and `allOf`

`anyOf` and `allOf` are both treated as composite composition inputs.

The generator tries to flatten them into a single object-like model when possible. If composition cannot be represented safely as one object type, it falls back to `Object`.

This is a pragmatic Java mapping choice, not full JSON Schema semantics.

### Mixed composition forms

Mixing composition kinds on one schema (for example `allOf` + `oneOf`, or `allOf` + `anyOf`) is not supported as a precise model. The schema is mapped to a generic fallback and a warning is logged.

### `not`

`not` is currently not modeled as a dedicated Java type strategy.

## OpenAPI Version Handling

- OpenAPI `3.0` and `3.1` are supported.
- Other versions are rejected.

Nullability behavior differs by spec version:

- OAS 3.0: nullability uses `nullable: true`.
- OAS 3.1: nullability uses schema types (`type`/`types` including `null`, or null-only variants in `oneOf`/`anyOf`).

## Explicit Non-Goals

The generator intentionally does not generate cross-cutting runtime concerns such as:

- security configuration
- logging configuration
- validation frameworks

Those concerns are expected to be handled by Spring and application code.

## Practical Limits and Failure Modes

- The generator does not pre-validate Java/OS limits (identifier/path length, etc.). You may need to adjust the OpenAPI spec if generated code does not compile on your platform.
- Missing component parameter refs fail generation.
- Multiple conflicting 2xx body schemas for one operation fail generation.

## Guidance For New Users

To get predictable output:

1. Provide stable `operationId` values.
2. Use tags intentionally; they define generated client interfaces.
3. Prefer a single successful response body schema per operation.
4. Prefer component schemas (`#/components/schemas/...`) for important reusable models.
5. Avoid mixing composition forms in one schema.
6. Use optional + nullable only when you need explicit-null semantics (this triggers `NullWrapper<T>`).
